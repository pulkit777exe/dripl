## 1. AI Text-to-Diagram

### Current Flow
```
User clicks "Text to Diagram (AI)" → AIGenerateModal opens
→ User types prompt → POST /api/ai/generate
→ Gemini API call → Response parsed → Elements added to store
→ CustomEvent dispatched → Modal closes
```

### BUG 1.1 (CRITICAL): `dripl:fit-elements` Event Has No Listener ✅ FIXED
**File:** `apps/dripl-app/components/canvas/AIGenerateModal.tsx:75-79`

**Problem:** After AI generation, the modal dispatches `new CustomEvent('dripl:fit-elements', { detail: { elementIds } })` but **no component listens for this event**. Generated elements are added to the store at default coordinates (typically near x:100, y:100) but the viewport never pans to show them.

**Fix:** In `RoughCanvas.tsx`, add an event listener for `dripl:fit-elements` that calls `fitElementsToScreen(elementIds)` from `useCanvasViewport`. Register in a `useEffect` and clean up on unmount.

**Status:** ✅ Added useEffect listener at line 276-284

### BUG 1.2 (CRITICAL): Arrow/Line Points Format Mismatch ✅ FIXED
**File:** `apps/dripl-app/app/api/ai/generate/route.ts:25` (system prompt)
**File:** `packages/common/src/schemas.ts:3-6` (PointSchema)

**Problem:** The AI system prompt instructs Gemini to return points as `[[x1, y1], [x2, y2]]` (nested arrays). But `PointSchema` expects `{ x: number, y: number }` objects. Every arrow, line, and freedraw element generated by the AI **fails validation and is silently filtered out**. Users only see rectangles, ellipses, diamonds, and text.

**Fix:** Update the system prompt to instruct the AI to return `{ x, y }` objects instead of `[x, y]` arrays. OR add a transformation step in the route handler that converts `[x, y]` to `{ x, y }` before validation.

**Status:** ✅ Updated system prompt + added normalization step (lines 178-190)

### BUG 1.3 (MODERATE): Zero-Value Coercion ✅ FIXED
**File:** `apps/dripl-app/app/api/ai/generate/route.ts:185-186`

**Problem:** `Number(el.roughness) || 1` coerces `0` to `1` because `0` is falsy. Valid zero-values are overridden.

**Fix:** Replace `|| 1` with `?? 1` (nullish coalescing) for all numeric defaults.

**Status:** ✅ Changed all numeric defaults to use nullish coalescing (lines 192-201)

### BUG 1.4 (MODERATE): No `fillColor` Passthrough
**File:** `apps/dripl-app/app/api/ai/generate/route.ts:169-194`

**Problem:** The element mapping step sets `backgroundColor` but never passes through `fillColor`. AI-specified fill styling is lost.

### BUG 1.5 (MINOR): Silent Filtering Without Feedback
**File:** `apps/dripl-app/app/api/ai/generate/route.ts:196-199`

**Problem:** Elements failing `DriplElementSchema.safeParse()` are silently removed. If all arrows fail, the user sees shapes with no connecting lines and no explanation.

### INCOMPLETE 1.6: No Collaboration Broadcast
**Problem:** After `addElements()`, the store mutates locally but `useCollaboration` differential sync may not pick up the change in a collaborative room.

### INCOMPLETE 1.7: No Streaming / Progress
**Problem:** The Gemini API call is awaited in full with no progress feedback beyond "Generating...". Complex diagrams can take several seconds.

## 3. Zoom & Pan Behavior

### ISSUE 3.1 (HIGH): No Zoom Smoothing/Animation ✅ FIXED
**File:** `apps/dripl-app/lib/store/canvasSlice.ts:417`

Zoom changes are instant. Excalidraw interpolates from current to target zoom over ~150ms with an easing curve. This makes Dripl feel "jerky" especially on trackpad pinch-to-zoom.

**Fix:** Implement smooth zoom animation using `requestAnimationFrame` interpolation. The `AnimationController.ts` already exists and can be extended.

**Status:** ✅ Added smoothZoom function to zoomUtils.ts using AnimationController

### ISSUE 3.2 (HIGH): Inconsistent Zoom Limits ✅ FIXED
| Location | Max Zoom |
|---|---|
| `canvasSlice.ts:417` | 20 |
| `RoughCanvas.tsx:1145` | 20 |
| `useCanvasPointerEvents.ts:485` | 20 |
| `CanvasControls.tsx:14-15` | **20** |
| `utils/zoomUtils.ts:11-13` | **20** |

UI buttons cap at 5x while wheel/pinch allow 20x.

**Fix:** Standardize all zoom limits to `[0.1, 20]`.

**Status:** ✅ Updated all zoom utilities to use [0.1, 20] range

### ISSUE 3.3 (HIGH): Inconsistent Zoom Factors ✅ FIXED
| Input Method | Formula | File |
|---|---|---|
| UI +/- buttons | `zoom * 1.1` / `zoom / 1.1` (multiplicative) | `CanvasControls.tsx:14-15` |

**Fix:** Use multiplicative factors everywhere for consistent feel.

**Status:** ✅ Updated CanvasControls.tsx and calculateZoom to use multiplicative factors

### ISSUE 3.4 (MODERATE): No `deltaMode` Normalization ✅ FIXED
**File:** `apps/dripl-app/components/canvas/RoughCanvas.tsx:1084-1160`

The wheel handler does not check `e.deltaMode`. On Firefox, trackpad events may use `DOM_DELTA_LINE` instead of `DOM_DELTA_PIXEL`, causing different zoom/pan speeds across browsers.

### ISSUE 3.5 (MODERATE): Trackpad Detection Is Fragile
**File:** `apps/dripl-app/components/canvas/RoughCanvas.tsx:1079-1097`

Uses a 150ms timeout heuristic that can misfire on mice with horizontal scroll wheels. The timeout may be too short for slow scrolling.

## 7. Open-Source Patterns to Extract

> Patterns from Excalidraw, tldraw, Fabric.js, Konva.js, React Flow, Mermaid.js, Yjs, and Automerge that can be directly adopted or adapted for Dripl.

---

### 7.1 Arrow Binding — Adopt tldraw's Model

tldraw has the most complete binding system. Bindings are **first-class entities** separate from shapes:

```typescript
// tldraw pattern — bindings as separate records
type TLArrowBinding = TLBaseBinding<"arrow", {
  terminal: "start" | "end",
  normalizedAnchor: { x: number, y: number },  // 0-1 normalized position on target
  isExact: boolean,    // pass through shape to anchor point
  isPrecise: boolean,  // respect normalizedAnchor vs snap to center
  snap: "center" | "edge-point" | "edge" | "none"
}>
```

**Key lifecycle hooks to implement:**
- `onAfterChangeFromShape` — when arrow moves, update binding
- `onAfterChangeToShape` — when target shape moves/resizes, update arrow endpoint
- `onBeforeIsolateFromShape` — "bake in" binding state before deletion/copy

**Excalidraw's approach** (simpler): Cast ray from arrow endpoint to shape center, find intersection with shape boundary, sort by distance. Uses `intersectElementWithLineSegment` with custom intersectors.

**Recommendation for Dripl:** Use tldraw's normalized anchor approach since the `NormalizedBinding` type already exists in `packages/common/src/types/element.ts:114-124`. Wire up the existing `recalculateBinding()` from `arrow-routing.ts`.

---

### 7.2 Rendering Pipeline — Adopt Excalidraw's Two-Canvas Architecture

Excalidraw separates rendering into two stacked canvases:

| Canvas | Purpose | Refresh Rate |
|--------|---------|--------------|
| **StaticCanvas** | Document elements (shapes, text, images) | On element/viewport change |
| **InteractiveCanvas** | Selection handles, cursors, snap lines | 60fps animation loop |

**Dripl already has this pattern** (`StaticCanvas.tsx` + `InteractiveCanvas.tsx`). But can improve:

1. **Per-element canvas caching** (Excalidraw pattern): Each element gets an off-screen `<canvas>` cached by `element.version` + zoom level. Avoids re-computing Rough.js shapes every frame.
2. **Dirty flag system** (Konva pattern): Only re-render elements that changed. Konva's `batchDraw()` consolidates multiple property changes into single `requestAnimationFrame`.
3. **Viewport culling via spatial index**: Excalidraw uses RBush to find visible elements, hides off-screen shapes with `display: none`. Dripl has RBush but doesn't use it for culling.

---

### 7.3 State Management — Adopt Zustand + Immer Pattern

React Flow and tldraw both use composable stores with selector-based subscriptions:

```typescript
// React Flow pattern
const store = createEditorStore()({
  sceneSlice: { nodes, edges, addNode, removeNode },
  selectionSlice: { selectedIds, selectNode, clearSelection },
  historySlice: { undo, redo, pushHistory },
  viewportSlice: { zoom, panX, panY, setZoom },
  uiSlice: { activeTool, setActiveTool },
});
```

**Key patterns to adopt:**
- **Selector-based subscriptions**: Components only re-render when their specific slice changes (Dripl already does this with `useShallow`)
- **Batch operations**: `startBatch()` / `endBatch()` for single undo points during multi-step operations
- **Transaction batching**: tldraw's `transact()` groups changes atomically with rollback support
- **History as deltas**: Excalidraw's `Store` computes deltas between snapshots, `History` consumes deltas. Different consumers (collaboration, storage) listen to increments independently.

---

### 7.4 Collaboration — Adopt Yjs CRDT Protocol

**Current Dripl:** Full-state broadcast or diff-based (scene-delta). No conflict resolution beyond version guards.

**Yjs approach** (production-grade, used by Excalidraw):
- Binary state vector exchange: Client sends what it has, server replies with only missing ops
- ~27 bytes per character append (vs full JSON re-serialization)
- Awareness channel for cursors/presence (15s heartbeat, 30s timeout)
- `Y.UndoManager` scoped to local user's client-id
- Persistence via `y-indexeddb` with snapshot compaction every ~1,000 updates

**Implementation path:**
1. Replace element array with `Y.Array` in room state
2. Use `y-websocket` provider for transport
3. Add awareness protocol for cursors
4. Keep existing version guards as fallback

---

### 7.5 Text-to-Diagram — Adopt Mermaid.js Parser Pattern

Mermaid.js uses a grammar-based approach:
1. **Grammar definitions** per diagram type (flowchart, sequence, etc.)
2. **Langium parser** (formerly Jison) → AST
3. **Renderer** converts AST to SVG elements

**For Dripl's AI approach:**
- Keep the Gemini API for natural language → structured output
- But add a **validation/rendering pipeline** similar to Mermaid:
  1. Parse AI response into typed elements
  2. Validate against schema (already done with Zod)
  3. **Auto-layout** elements if they overlap (use dagre or elkjs for graph layout)
  4. Render with proper bindings (arrows to shapes)

**Key library:** `dagre` (graph layout) or `elkjs` (layered graph layout) for automatic positioning of AI-generated diagrams.

---

### 7.6 Shape System — Adopt tldraw's Data/Behavior Separation

tldraw separates shape data from behavior:

```typescript
// Shape record = pure JSON in store
interface TLShape {
  id: string;
  type: string;
  x: number;
  y: number;
  props: Record<string, any>;
}

// ShapeUtil = behavior (rendering, geometry, interactions)
class RectangleShapeUtil extends ShapeUtil {
  getDefaultProps() { return { w: 100, h: 100 }; }
  getGeometry() { return new Rectangle2d(this.shape.props); }
  component() { return <RectangleShape shape={this.shape} />; }
  onResize(info) { /* resize logic */ }
  onRotate(info) { /* rotate logic */ }
}
```

**Benefits:**
- Shape types are extensible without modifying core
- Geometry classes enable hit testing, bounds calculation, binding
- Lifecycle hooks (`onBeforeCreate`, `onBeforeUpdate`, `onResize`, `onRotate`) centralize mutation logic

**For Dripl:** The element factory pattern in `packages/element/src/` already partially does this. Could formalize by creating `ShapeUtil` classes for each element type.

### 7.7 Performance — Adopt Konva's Optimization Techniques

| Technique | Source | Impact |
|-----------|--------|--------|
| **Node caching** (offscreen canvas) | Konva | 10-100x for complex shapes |
| **`batchDraw()`** | Konva | 2-10x by consolidating RAF calls |
| **`layer.listening(false)`** | Konva | 2-5x by skipping hit graph |
| **Transform caching** | Konva | 1.5-3x by caching matrices |
| **Dirty flag system** | Fabric, Konva | Avoids unnecessary redraws |
| **`perfectDrawEnabled(false)`** | Konva | Skip buffer canvas for simple shapes |

**Dripl-specific optimizations to add:**
1. **Element canvas cache version check** (partially done in TODOS #37): Invalidate cache only when `element.version` changes
2. **Shape cache for Rough.js** (done in TODOS): Pre-compute `Drawable` objects
3. **Spatial index for viewport culling** (RBush exists, not used for culling)
4. **ThrottleRAF for renders** (Excalidraw pattern): Throttle static scene renders to animation frame

### 7.8 Zoom/Pan — Adopt Excalidraw's Smooth Interpolation

Excalidraw's zoom implementation:

```typescript
// Keep point fixed while zooming
getStateForZoom({ viewportX, viewportY, nextZoom }, appState): Partial<AppState> {
  const sceneX = (viewportX - appState.scrollX) / appState.zoom.value;
  const sceneY = (viewportY - appState.scrollY) / appState.zoom.value;
  return {
    zoom: { value: nextZoom },
    scrollX: viewportX - sceneX * nextZoom,
    scrollY: viewportY - sceneY * nextZoom,
  };
}

// Zoom to fit bounds with animation
zoomToFitBounds({ bounds, appState, fitToViewport }): Partial<AppState>
```

**Key patterns:**
- **Cursor-anchored zoom**: Zoom keeps the point under cursor fixed (Dripl already does this)
- **Smooth interpolation**: Animate from current to target over ~150ms with easing
- **Auto-pan while dragging**: When dragging elements near viewport edges, canvas auto-pans using RAF loop
- **`deltaMode` normalization**: Account for `DOM_DELTA_PIXEL`, `DOM_DELTA_LINE`, `DOM_DELTA_PAGE`
